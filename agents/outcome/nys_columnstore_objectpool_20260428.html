4.1 — XList::Delete (xlist.inl:357) — The Hot Spinlock

template <class TElem>
void inline XList<TElem>::Delete(__in TElem* const el)
{
    m_lock.Get();              // ← SPINLOCK ACQUIRE (type 27)

    // Invalidate all iterators — O(N) walk of enumerator list
    TEnumerators::EnumeratorType iter(&m_enumerators);
    for (EnumeratorType* pEnum = iter.GetNext(nullptr);
         pEnum != nullptr;
         pEnum = iter.GetNext(pEnum))
    {
        pEnum->Invalidate(el);
    }

    TElements::Delete(el);     // Remove element from linked list

    m_lock.Release();          // ← SPINLOCK RELEASE
}

⚠ The spinlock is held while iterating ALL enumerators. Under heavy concurrent access, this creates a serialization point across all threads accessing the same cache partition.

Why 0% CPU? The threads have already exceeded the spin count and fallen into SpinlockBase::Sleep() which calls SleepEx() — a Windows kernel sleep. They are not burning CPU spinning; they are blocked waiting. The non-yielding detection triggers because the scheduler hasn't yielded within the timeout, even though no CPU work is being done.

6. Root Cause Analysis

Root Cause: Spinlock Contention on Cache Simulation Descriptor Lists

The non-yielding condition is caused by heavy spinlock contention on the XList<ProbCostData> data structure within the SOS CacheStore's probabilistic cache simulation algorithm.

Mechanism

  1. Multiple parallel threads execute a SELECT INTO with hash join probing a columnstore index
  2. Each thread iterates through row groups via RowGroupManager::GetNextRowGroup
  3. For each row group, column segments are retrieved via ColumnStoreObjectPool::RetrieveObjectAndFix
  4. On cache miss, the object is inserted into the CacheStore via CacheUserDataExclusiveWithControl
  5. The cache store's probabilistic algorithm promotes simulated entries to actual entries via ConvertEntryToActual → SimulationHit
  6. This calls XList<ProbCostData>::Delete which acquires a per-partition spinlock
  7. With 17+ threads all hitting the same cache partition, the spinlock becomes a serialization bottleneck
  8. Threads spin, exceed the spin count, and fall into SleepEx()
  9. The sleep duration exceeds the non-yielding threshold (typically 15-60 seconds)

Why This Callstack Is Different From Prior Fixes

  • PR 1298481 fixed a race condition in XList Invalidate — but the contention remains
  • The spinlock type changed from 28 to 27 (possibly due to spinlock class renumbering), but the ID (268435714) is the same
  • The underlying problem is insufficient partitioning of the cache store's simulation lists

9. Fix Recommendation

Recommended Fix: Increase Cache Store Partitioning for ColumnStoreObjectPool

The core problem is that the XList<ProbCostData> simulation lists are partitioned insufficiently for high-concurrency columnstore workloads. The fix surface involves:

Option A: Dynamic PMO Partitioning (Strategic — aligns with Bug 4840571)

  • Increase the number of partitions in the TPartitionedDescriptorList for the ColumnStoreObjectPool cache store
  • Use contention-based dynamic partitioning (as noted in Bug 4840571 comments)
  • This spreads threads across more partitions, reducing per-partition contention

Option B: Reduce Hold Time on XList Spinlock

  • The XList::Delete method holds the spinlock while iterating ALL enumerators for invalidation
  • Consider deferring enumerator invalidation or using a lock-free invalidation mechanism
  • This was partially addressed by PR 1298481 but the contention remains

Fix Checklist

  • Yield interval: use power-of-2 count for compiler optimization (N/A — spinlock issue, not loop yield)
  • YieldAndCheckForAbort vs plain Yield: Not applicable — issue is spinlock contention, not missing yield
  • Verify yield does NOT happen while holding latches/locks
  • Check engine patterns for consistent partition count choices
  • Identify ALL entry points: XList::Delete, XList::Insert, XListEnumerator::GetNext — all 3 acquire the same spinlock
