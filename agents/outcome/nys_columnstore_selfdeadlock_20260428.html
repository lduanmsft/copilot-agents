6. PR 1298481 Fix Analysis

Fix Approach: Deferred Release

PR 1298481 by Keiichiro Koga, merged 2024-03-19. Core changes in xlist.inl.

Before fix: Delete() calls Invalidate() while holding the spinlock. Invalidate() directly calls Release() — if this triggers a recursive deletion, it attempts to acquire the already-held spinlock.

After fix:

// xlist.inl (AFTER FIX) — Delete() function
void XList<TElem>::Delete(__in TElem* const el)
{
    TElem* deferredReleaseObject = nullptr;    // ← NEW: deferred release pointer
    UINT deferredReleaseCount = 0;             // ← NEW: deferred release count

    {
        Holder lockScope = m_lock.GetIntoHolder();  // Acquire spinlock (RAII scope)

        AddRecord(XLO_Delete, el);

        // Invalidate all iterators — but NO LONGER releases immediately
        for (EnumeratorType* pEnum = ...; pEnum != nullptr; ...)
        {
            pEnum->Invalidate(
                el,
                &deferredReleaseObject,        // ← NEW: collect objects to release
                deferredReleaseCount);          // ← NEW: count
        }

        TElements::Delete(el);
    }   // ← ★ Spinlock RELEASED here (RAII lockScope destructor)

    // ★★★ KEY FIX ★★★
    // Release operations moved OUTSIDE the spinlock scope!
    // Even if Release triggers RemoveAndDestroy → recursive Delete(),
    // it will NOT deadlock because the spinlock is no longer held
    for (UINT i = 0; i < deferredReleaseCount; i++)
    {
        deferredReleaseObject->Release();      // ← SAFE: spinlock is NOT held
    }
    deferredReleaseObject = nullptr;
}

7. Root Cause

Spinlock Self-Deadlock in XListEnumerator::Invalidate

A race condition in XListEnumerator::Invalidate in the SOS cache framework (xlist.inl). The XList data structure used by the probabilistic cache simulation algorithm has a flaw where Invalidate releases a reference on the base object prematurely. When the same thread re-enters through the cache store's release path (Fix → Unfix → ReleaseUserData → SimulationInsertion → XList operations), the non-reentrant Spinlock<27,1,268435714> is acquired twice by the same thread, causing a permanent self-deadlock.

PG's confirmed conclusion: "The thread is deadlocked on itself."

What the Fix Does

  1. Removes premature reference release in XListEnumerator::Invalidate
  2. Keeps the reference added in GetPrevElem until the iteration advances or finishes
  3. Introduces deferred release in XList::Delete — previous base object references are released outside the spinlock scope
  4. Controlled by TF 16405 (XListFix feature switch), marked as Reserved (default-on) in latest branches
