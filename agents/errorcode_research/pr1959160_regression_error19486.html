🔍 PR 1959160 Regression Analysis — Error 19486

Analysis of how "Certificate Rotation Without SQL Restart" (CU24) could cause HADR AG listener TCP port conflict (Error 19486)

Executive Summary

PR 1959160 introduces "Certificate Rotation Without SQL Restart" for SQL Server 2022 CU24. While the primary feature is gated behind TF 11842 (disabled by default), the PR also modifies the SNI_CONSUMER_INFO struct used by all SNI listeners — including HADR AG listeners. Two bugs in these unconditional changes can cause Error 19486 (TCP port already in use) during AG listener operations:

  1. CRITICAL Broken duplicate listener detection — FMatchNonProviderSpecific() always returns FALSE when wszInstanceName is NULL, defeating duplicate-listener protection and causing fresh port binds that fail with WSAEADDRINUSE
  2. HIGH dwRet overwrite in InitializeListener Exit — if TF 11842 is enabled, a cert-monitoring failure overwrites a successful listener init result, causing SSL listener setup to report failure

2. Error 19486 Background

Error 19486 = HADR_ERROR2 (194) × 100 + HADR_DDL_LISTENER_TCP_PORT_IN_USE_FAILURE (86)

Message: "TCP provider failed to listen on [...]. TCP port is already in use."

Raised in HadrAGListenerNotifier::ReportFailureToUser() when the underlying TDS/SNI layer fails to bind a TCP port because it's already in use (WSAEADDRINUSE / Win32 error 10048) or access denied (WSAEACCES / 10013).

Call flow: ALTER AVAILABILITY GROUP DDL → WaitForListenersSynchronization() → TDS synchronizes listeners → SNI Tcp::InitializeListener binds port → bind() fails → Error 19486

3. Bug #1: Broken Duplicate Listener Detection

CRITICAL sni.hpp FMatchNonProviderSpecific() Lines 842-843

What Changed

The PR added wszInstanceName to the SNI_CONSUMER_INFO equality check, making it an AND condition with all other fields:

// sni.hpp — FMatchNonProviderSpecific()
bool FMatchNonProviderSpecific(const Sni_Consumer_Info *pOtherConsumerInfo) const
{
    if (this->DefaultUserDataLength == pOtherConsumerInfo->DefaultUserDataLength &&
        this->ConsumerKey == pOtherConsumerInfo->ConsumerKey &&
        this->fnReadComp == pOtherConsumerInfo->fnReadComp &&
        ... 12 more field comparisons ...
        this->eConsumerType == pOtherConsumerInfo->eConsumerType &&
        // ↓↓↓ NEW — Added by PR 1959160 ↓↓↓
        (this->wszInstanceName != NULL && pOtherConsumerInfo->wszInstanceName != NULL &&
         0 == _wcsicmp(this->wszInstanceName, pOtherConsumerInfo->wszInstanceName)))
    {
        return true;   // Match → listener is a duplicate
    }
    return false;      // No match → treat as new listener → bind port fresh
};

The Bug

If either wszInstanceName is NULL, the entire expression evaluates to FALSE — regardless of whether all 15 other fields match perfectly. This means FMatchNonProviderSpecific() never returns TRUE when wszInstanceName is NULL on either side.

Why wszInstanceName Is NULL in AG Listener Paths

The HADR dynamic listener code path in listenhelper.cpp (L3693-3714) does not set wszInstanceName:

// listenhelper.cpp — AddDynamicListener() — HADR path
MEMZERO (&snici, sizeof(SNI_CONSUMER_INFO));   // wszInstanceName = NULL (zeroed)
snici.DefaultUserDataLength = pSls->DefaultUserDataLength;
snici.fnReadComp   = pSls->fnReadComp;
... other fields set ...
snici.eConsumerType = SNI_CONSUMER_TYPE_TSQL;
// ↑ NO snici.wszInstanceName assignment! Stays NULL from MEMZERO

// COMPARE with static listener path (SNIStartListening L3371):
snici.wszInstanceName = pSls->wszInstanceName;  // ← Only set in static path

Impact — How This Causes Error 19486

The FMatchNonProviderSpecific() function is called in SNIInitializeListener() (sni.cpp L2936) to detect duplicate listeners:

// sni.cpp — SNIInitializeListener()
for (SNI_Listener *pListenerToCompare = g_pListenerList; ...)
{
    if (pListenerToCompare->ConsumerInfo.FMatchNonProviderSpecific(pConsumerInfo))
    {
        // Check TCP provider-specific info
        fIsDuplicate = Tcp::CheckForDuplicateListener(...);
    }
}

// Later...
case TCP_PROV:
    dwError = Tcp::InitializeListener(hSNIListener, ..., fIsDuplicate);

// In Tcp::InitializeListener() (tcp.cpp L6297):
if (fIsDuplicate) {
    return Tcp::InitializeDuplicateListener(...);  // Safe: reuses existing socket
}
// else: creates NEW socket → bind() → WSAEADDRINUSE if port taken!
Regression Flow:
AG DDL → AddDynamicListener (wszInstanceName=NULL)
↓
FMatchNonProviderSpecific() → returns FALSE (because NULL check fails)
↓
fIsDuplicate = false (skips duplicate listener check entirely)
↓
Tcp::InitializeListener() tries to bind port fresh
↓
Port already bound by existing listener → WSAEADDRINUSE (10048)
↓
Error 19486: TCP port is already in use

Correct Fix

The NULL case should be treated as "match" (equivalent to both being NULL = same instance context):

// FIXED: Handle NULL wszInstanceName correctly
(this->wszInstanceName == pOtherConsumerInfo->wszInstanceName ||   // Both NULL → match
 (this->wszInstanceName != NULL && pOtherConsumerInfo->wszInstanceName != NULL &&
  0 == _wcsicmp(this->wszInstanceName, pOtherConsumerInfo->wszInstanceName)))

OR: Set wszInstanceName in the HADR dynamic listener path (listenhelper.cpp L3702):

snici.wszInstanceName = pSls->wszInstanceName;  // Add this line at L3702

4. Bug #2: dwRet Overwrite in SSL InitializeListener

HIGH SNI_SslProvider.cpp Ssl::InitializeListener() Lines 4141-4172

Condition: Only triggers when TF 11842 is enabled

The Bug

// SNI_SslProvider.cpp — Ssl::InitializeListener() Exit section
Exit:
    // ... cleanup ...
    
    if (ERROR_SUCCESS == dwRet)   // Listener init succeeded!
    {
        if (TF(TRCFLG_RETAIL_ENABLE_SERVER_CERTIFICATE_ROTATION_MONITORING) &&
            SOS_Node::GetCurrent()->IsRootMemoryNode())
        {
            dwRet = StartSslCertificateRegistryMonitoring(s_wszInstanceName);
            // ↑ BUG: Overwrites dwRet with monitoring result!

            if (ERROR_SUCCESS != dwRet)
            {
                // "Note: We intentionally do not fail the initialization..."
                // But the code DOES return dwRet (which is now an error)!
                SNI_TRACE_WARNING(...);
            }
        }
    }
    
    return dwRet;  // Returns monitoring error, NOT the listener init result!

Impact

When TF 11842 is enabled and cert registry monitoring fails to start (e.g., registry key doesn't exist, access denied, etc.), the SSL listener initialization is reported as failed even though it succeeded. This can cascade to AG listener setup failure.

Correct Fix

DWORD dwMonitorRet = StartSslCertificateRegistryMonitoring(s_wszInstanceName);
if (ERROR_SUCCESS != dwMonitorRet) {
    SNI_TRACE_WARNING(...);
    // Don't overwrite dwRet — monitoring is optional
}

5. Bug #3: Missing wszInstanceName in HADR Dynamic Listener Path

MEDIUM listenhelper.cpp AddDynamicListener() Line 3693-3702

What's Missing

The static listener path (SNIStartListening) was updated to set snici.wszInstanceName (line 3371), but the HADR dynamic listener path (AddDynamicListener) was not updated:

Code PathSets wszInstanceName?Used By
SNIStartListening() — L3371✅ Yes (pSls->wszInstanceName)Static listeners (startup)
AddDynamicListener() — L3693-3702❌ No (MEMZERO → NULL)HADR AG listeners

This inconsistency is the root cause of Bug #1. Even if Bug #1's NULL-comparison logic is fixed, the HADR path should still set wszInstanceName to maintain consistency with the SNI_CONSUMER_INFO contract.

7. Reproduction Scenario

  1. Install SQL Server 2022 CU24 (which includes PR 1959160)
  2. Create an Availability Group with a TCP listener on a specific port (e.g., 5022)
  3. The static listener is initialized via SNIStartListening → wszInstanceName is set
  4. An AG DDL operation triggers AddDynamicListener for the same port
  5. MEMZERO at L3693 zeroes wszInstanceName to NULL
  6. FMatchNonProviderSpecific() compares:
    • Existing listener: wszInstanceName = "MSSQLSERVER" (or instance name)
    • New listener: wszInstanceName = NULL
    • NULL check: NULL != NULL → condition is FALSE
    • FMatchNonProviderSpecific returns FALSE → fIsDuplicate = FALSE
  7. Tcp::InitializeListener attempts fresh bind() on the port
  8. Port already in use → WSAEADDRINUSE (10048) → Error 19486

9. Recommendations

Immediate Workaround

  • Avoid CU24 if experiencing Error 19486 on HADR AG listener operations
  • If already on CU24, avoid AG DDL operations (ALTER AVAILABILITY GROUP) that modify listeners until a fix is available
  • Do NOT enable TF 11842 until Bug #2 is also fixed

Required Code Fixes

  1. [CRITICAL] Fix FMatchNonProviderSpecific() in sni.hpp: Handle NULL wszInstanceName as a valid match condition
  2. [HIGH] Fix dwRet overwrite in SNI_SslProvider.cpp InitializeListener(): Use a separate variable for monitoring result
  3. [MEDIUM] Set wszInstanceName in HADR AddDynamicListener() path in listenhelper.cpp
