⚠ Primary: Server Resource Pressure
  • Authentication completed in 2ms — the server accepted credentials instantly
  • But the server took 14,463ms to process the post-login SET commands and database context switch
  • This means SQL Server's worker threads were saturated or blocked handling other requests

  • Increase connection timeout: Add Connection Timeout=30 (or 60) to connection string

✅ What's Correct

SPN/Kerberos Recommendation & PREEMPTIVE_OS_AUTHORIZATIONOPS Are Valid
  • PREEMPTIVE_OS_AUTHORIZATIONOPS is real and relevant. From secnt.cpp, it wraps calls to:
    • LookupAccountNameInternal() — resolves Windows account name → SID
    • LookupAccountSidInternal() — resolves SID → account name (called per group)
  • These happen in CWindowsSecurityPrimaryInfo::GetNtGroupsViaAuthZ() which enumerates all AD group memberships via:
    AuthzInitializeContextFromSid → AuthzGetInformationFromContext → per-group LookupAccountSidInternal
  • With NTLM, group membership resolution requires Domain Controller round-trips. Kerberos tickets carry group info, reducing these calls.
  • Testing with SQL authentication is a valid isolation step to rule out Windows auth as the cause.

⚠️ What's Technically Imprecise

1. "delay during Windows authentication" — should be "authorization"

PREEMPTIVE_OS_AUTHORIZATIONOPS is about authorization (group membership enumeration, permission checks), NOT authentication. Authentication (SSPI handshake) happens during the Login phase. The wait type name itself says "AUTHORIZATION", not "AUTHENTICATION".

2. Group enumeration timing vs. client-reported phases

From tds_c_events.cpp, the server login flow is:

process_login()
  → credential validation (SSPI)
  → fnprocess_login_finish()
      → authorization + group enumeration  ← PREEMPTIVE_OS_AUTHORIZATIONOPS here
      → sends LOGINACK token to client
      → UnSetInLogin()
  → client receives LOGINACK → client's "Post-Login" starts

The group enumeration happens BEFORE the server sends LOGINACK. From the client's perspective, this delay should appear in [Login] authentication time, not [Post-Login].

3. What actually causes Post-Login delays with Windows auth

Post-Login delays with Windows authentication can occur when:

  • The client sends USE [database] during post-login, and the server does database-level permission checks which trigger additional LookupAccountSid calls under PREEMPTIVE_OS_AUTHORIZATIONOPS
  • Login triggers that reference Windows principals or perform AD lookups
  • sp_reset_connection on pooled connections that need to re-validate permissions
  • Server resource pressure preventing timely processing of initial SET commands (the case in our original error)
