1️⃣ Overview — How TDE with AKV Works

When Transparent Data Encryption (TDE) is configured with a Customer-Managed Key (CMK) stored in Azure Key Vault (AKV), the Managed Instance requires continuous access to the TDE protector key to keep databases online.

🔑 Required Key Vault Permissions
The Managed Instance's managed identity must have the following permissions on the key:
  • Get — retrieve the key
  • WrapKey — encrypt the Database Encryption Key (DEK)
  • UnwrapKey — decrypt the DEK

If the MI loses access to the AKV key for any reason, databases encrypted with that key will eventually become Inaccessible — the only action allowed on an Inaccessible database is deleting it (until the key access is restored and revalidated).

2️⃣ Key Loss Scenarios (5 Categories)

There are five major categories of AKV key loss. Each produces different Azure Key Vault error codes and triggers different recovery timelines.

🚫 Scenario 1: Missing Permissions (Error 401)

401 AzureKeyVaultMissingPermissions — The server is missing required permissions on the Azure Key Vault.

Cause
  • Key Vault Get, WrapKey, or UnwrapKey permissions revoked from the MI's managed identity
  • Key Vault access policy deleted
  • RBAC role assignment removed (if using RBAC-based access model)
Error Type4XX Error → Database becomes Inaccessible after 30 minutes
How to Fix AKV
  1. Navigate to Key Vault → Access Policies (or IAM for RBAC model)
  2. Find the MI's managed identity (system-assigned or user-assigned)
  3. Re-grant: Get, WrapKey, UnwrapKey permissions
  4. If using RBAC: assign Key Vault Crypto Service Encryption User role

🗑️ Scenario 2: Key Deleted or Disabled (Error 404 / 409)

404 ServerKeyNotFound — The requested server key was not found on the current subscription.

409 ServerKeyDoesNotExists — The server key does not exist.

Cause
  • The AKV key was accidentally deleted
  • The key was disabled
  • The key expired
Error Type4XX Error → Inaccessible after 30 minutes
How to Fix AKV
  1. If deleted (with soft-delete enabled): Recover the key using az keyvault key recover or Portal → Key Vault → Managed Deleted Keys → Recover
  2. If deleted (no soft-delete): Key is permanently lost — restore from backup key or re-create from key material backup
  3. If disabled: Enable the key in Key Vault → Keys → select key → Enable
  4. If expired: Extend the key expiration date

🏗️ Scenario 3: Key Vault Unavailable (Error 503)

503 AzureKeyVaultConnectionFailed — The operation could not be completed on the server because attempts to connect to Azure Key Vault have failed.

Cause
  • The Key Vault itself was accidentally deleted
  • Key Vault firewall blocks Microsoft services
  • Key Vault is in a different region experiencing an outage
Error Type4XX / 5XX Error → 30 min or 24-hour buffer (see Timeline)
How to Fix AKV
  1. If deleted (with soft-delete): Recover: az keyvault recover --name <vault-name>
  2. If firewall issue: Key Vault → Networking → enable "Allow trusted Microsoft services to bypass this firewall"
  3. If regional outage: Wait for resolution or use a backup key from another Key Vault

🪪 Scenario 4: Managed Identity Deleted (Error 401)

401 AzureKeyVaultNoServerIdentity — The server identity is not correctly configured on server.

Cause
  • The MI's system-assigned or user-assigned managed identity was accidentally deleted in Microsoft Entra ID
  • The MI was moved to a different subscription (identity broken)
  • The MI was moved to a different Microsoft Entra tenant
Error Type4XX Error → Inaccessible after 30 minutes
How to Fix AKV
  1. Re-assign a managed identity to the MI: 
    # PowerShell
Set-AzSqlServer -ResourceGroupName <RG> -ServerName <MI> -AssignIdentity

# Azure CLI
az sql server update --resource-group <RG> --name <MI> --assign_identity
  2. Re-add the new managed identity to Key Vault access policies with Get, WrapKey, UnwrapKey permissions
  3. If tenant was changed: redo the full identity + Key Vault configuration in the new tenant

🌐 Scenario 5: Network Connectivity Blocked (MI-Specific)

The Key Vault endpoint exists but cannot be reached from the Managed Instance subnet.

Cause
  • AKV exposed via Private Endpoint but the private IP is not allowed in the MI subnet's NSG outbound rules
  • Bad DNS resolution — AKV FQDN not resolved or resolves to an invalid IP
  • Route table misconfiguration or virtual appliance blocking port 443
Error Type5XX Error → 24-hour buffer before Inaccessible
How to Fix
  1. Test connectivity from MI: 
    -- Test from MI (TCP connectivity check)
-- Endpoint: <vault_name>.vault.azure.net
-- Port: 443
-- Expected: TcpTestSucceeded: True
  2. If TcpTestSucceeded: False:
    • Check NSG outbound rules — allow AKV IP on port 443
    • Check DNS resolution — verify AKV FQDN resolves to correct IP
    • Check route table and virtual appliance configuration
    • For Private Endpoint: ensure the private IP is in NSG outbound allow list

Scenario Summary

Scenario Error Code Error Type Time to Inaccessible
Permissions revoked 401 4XX 30 minutes
Key deleted/disabled/expired 404 / 409 4XX 30 minutes
Key Vault unavailable 503 5XX 24 hours
Managed identity deleted 401 4XX 30 minutes
Network connectivity blocked 503 (5XX) 5XX 24 hours

3️⃣ Timeline — What Happens After Key Loss

Path A: 4XX Errors (Permission / Key Issues) — 30-Minute Window

T + 0 min
MI loses access to AKV key. Cached encryption key (DEK) still in memory — database may still work briefly.
T + ~10 min
Database starts denying all new connections with error message.
T + 30 min
Database state changes to Inaccessible. Auto-heal cutoff point. Only allowed action: DELETE the database.
T + 30 min → ∞
If AKV fixed after this point → manual revalidation required via Azure Portal.

Path B: 5XX Errors (Network / Transient Issues) — 24-Hour Buffer

T + 0 min
MI cannot reach AKV endpoint. 24-hour buffer begins.
T + ~10 min
Database starts denying new connections.
T + 0 → 24 hours
Buffer period — if network is restored during this time, databases auto-recover. No action needed.
T + 24 hours
Database moves to Inaccessible state. Same manual recovery procedure required.
⚠️ Critical Window
For 4XX errors: You have 30 minutes to fix the AKV issue for automatic recovery.
For 5XX errors: You have 24 hours — databases auto-recover if network is restored within this window.

4️⃣ Step 1: Fix the AKV Issue

Before you can revalidate the key on the MI, you must first resolve the underlying AKV problem. Choose the fix based on your scenario:

Problem Detection Command Fix
Permissions revoked 
# Check current access policies
az keyvault show --name <vault> \
  --query "properties.accessPolicies"

# Re-grant permissions
az keyvault set-policy \
  --name <vault> \
  --object-id <MI-identity-object-id> \
  --key-permissions get wrapKey unwrapKey
Key deleted
(soft-delete on)		 
# List deleted keys
az keyvault key list-deleted \
  --vault-name <vault>

# Recover deleted key
az keyvault key recover \
  --vault-name <vault> \
  --name <key-name>
Key expired 
# Check key expiration
az keyvault key show \
  --vault-name <vault> \
  --name <key-name> \
  --query "attributes.expires"

# Extend expiration date
az keyvault key set-attributes \
  --vault-name <vault> \
  --name <key-name> \
  --expires "2027-12-31T23:59:59Z"
Key Vault deleted
(soft-delete on)		 
# List deleted vaults
az keyvault list-deleted

# Recover Key Vault
az keyvault recover --name <vault>
Firewall blocking Check Key Vault → Networking settings 
# Allow trusted Microsoft services
az keyvault update --name <vault> \
  --bypass AzureServices
Identity deleted 
# Check MI identity
az sql mi show --name <MI> \
  --resource-group <RG> \
  --query "identity"

# Re-assign identity
az sql mi update --name <MI> \
  --resource-group <RG> \
  --assign-identity

# Then re-add to Key Vault policies
az keyvault set-policy \
  --name <vault> \
  --object-id <new-identity-id> \
  --key-permissions get wrapKey unwrapKey
Network blocked
(MI-specific)		 Test TCP to <vault>.vault.azure.net:443 from MI
  • Fix NSG outbound rules
  • Fix DNS resolution
  • Fix route table / virtual appliance
  • Allow Private Endpoint IP in NSG

5️⃣ Step 2: Revalidate the Key on Managed Instance

After fixing the AKV issue, you must revalidate the TDE protector key on the Managed Instance to bring the database back online.

ℹ️ Pre-requisite
The AKV issue must be resolved first. The "Key revalidation" option only appears in the Portal when the server detects that the TDE protector key is inaccessible and the underlying issue has been fixed.

Method 1: Azure Portal (Recommended)

1
Navigate
Azure Portal → SQL Managed Instance resource
2
Open TDE
Security → Transparent Data Encryption
3
Revalidate
Click "Key revalidation" banner
4
Choose Option
"Retry existing key" or "Select backup key"
5
Wait
Database transitions to accessible (time depends on DB size)

Method 2: Azure CLI

az sql db tde key revalidate \
  --resource-group <ResourceGroupName> \
  --server <ServerName> \
  --database <DatabaseName>

Method 3: PowerShell

Invoke-AzSqlDatabaseTransparentDataEncryptionProtectorRevalidation `
  -ResourceGroupName <ResourceGroupName> `
  -ServerName <ServerName> `
  -DatabaseName <DatabaseName>

Method 4: REST API

POST https://management.azure.com/subscriptions/{subscriptionId}
     /resourceGroups/{resourceGroupName}
     /providers/Microsoft.Sql/servers/{serverName}
     /databases/{databaseName}
     /encryptionProtector/current/revalidate
     ?api-version=2022-08-01-preview
✅ Two Revalidation Options
  • Retry existing key — Re-establish access using the same key (use when you fixed the permission/network issue)
  • Select backup key — Configure a different key from your Key Vault (use when the original key is permanently lost)

6️⃣ Recovery Paths by Duration

AKV Outage Duration Recovery Method Estimated Time Data Loss?
< 30 minutes Automatic — database auto-heals, no action needed ~1 hour after AKV restored ❌ No
30 min – 24 hours Manual — Key revalidation via Portal / CLI / PowerShell / REST API Depends on database size ❌ No data loss, but settings lost
> 24 hours Manual — Same key revalidation procedure. Can take significant time Depends on database size (can be very long) ❌ No data loss, but all settings lost
5XX + < 24 hours Automatic — 24-hour buffer for network issues; database auto-recovers Automatic once network restored ❌ No
🚨 Key Permanently Lost (No Soft-Delete, No Backup)
If the key is permanently destroyed with no soft-delete recovery and no key material backup, the database cannot be recovered. The encrypted data is unrecoverable. This is why Key Vault soft-delete and purge protection should always be enabled.

7️⃣ What's Lost After Recovery (Manual Path)

Once a database is brought back online after the 30-minute auto-heal window, the following settings are permanently lost and must be manually reconfigured:

Category Lost Setting Action Required
High Availability Failover Group (FOG) configuration Re-create FOG, wait for secondary re-seeding
Backup Point-in-time restore (PITR) history ❌ Cannot recover — PITR history starts fresh
Backup Long-term retention (LTR) policy Re-configure LTR policy
Management Tags Re-apply all tags
Performance Elastic pool configuration Reconfigure
Performance Read scale Reconfigure
Performance Auto-pause settings Reconfigure
⚠️ Post-Recovery Checklist
After successful recovery, immediately validate all server-level and database-level settings. Microsoft recommends implementing a notification system (Azure Monitor + Action Groups) to detect key access loss within 30 minutes.

8️⃣ MI-Specific: Network Connectivity Troubleshooting

SQL Managed Instance has unique networking requirements for AKV access because it runs inside a virtual network subnet.

Common Network Causes

Issue Symptom Fix
Private Endpoint IP blocked by NSG AKV uses Private Endpoint, but MI subnet NSG doesn't allow outbound to the private IP on port 443 Add outbound rule allowing the AKV private IP on port 443
DNS resolution failure AKV FQDN doesn't resolve or resolves to wrong IP Check DNS settings, Private DNS Zone configuration
Route table misconfiguration Traffic to AKV routed through a virtual appliance that blocks/drops it Fix UDR (User Defined Route) to allow AKV traffic

How to Test Connectivity from MI

-- Test TCP connectivity from Managed Instance to AKV:
-- Endpoint: <vault_name>.vault.azure.net (without https://)
-- Port: 443
--
-- Expected results:
--   RemoteAddress: should exist and be the correct IP
--   TcpTestSucceeded: True
--
-- If TcpTestSucceeded: False → review NSG, DNS, route table

9️⃣ Best Practices & Prevention

# Practice Details
1 Enable Key Vault Soft-Delete & Purge Protection Prevents permanent key/vault deletion. Soft-delete is now enabled by default. Enable purge protection for additional safety.
2 Set up 30-minute alerting Use Azure Monitor Action Groups to detect AKV access loss within 30 minutes — this is the auto-heal window.
3 Rotate keys before expiration Use Azure Policy to enforce key rotation. Rotate well before expiration to avoid surprise outages.
4 Document all configurations Keep records of FOG config, PITR settings, tags, LTR policies — all are lost after manual recovery.
5 Use Infrastructure as Code (IaC) Maintain ARM/Bicep/Terraform templates for all MI settings so they can be quickly re-applied after recovery.
6 Test recovery procedures Periodically test the key revalidation workflow in a non-production environment.
7 Use geo-redundant Key Vault For Failover Groups, ensure primary and secondary MI each have their own AKV in their respective regions.
8 Audit Key Vault access Enable AKV diagnostic logging to detect unauthorized permission changes early.
