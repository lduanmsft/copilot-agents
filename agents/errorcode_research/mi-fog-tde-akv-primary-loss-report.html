Architecture Overview

This report analyzes a specific architecture where the Primary and Secondary Managed Instances each access their own separate Azure Key Vault for TDE (Transparent Data Encryption) with Customer-Managed Keys (CMK).

ℹ️ Key Architecture Requirement When separate AKVs are used for primary and secondary, the customer is responsible for keeping the key material consistent across both key vaults. The secondary must be able to decrypt data using the same key (or a copy of the key) from its own AKV.

Incident Timeline — What Happens After Primary Loses AKV Access

The following timeline is based on official Microsoft documentation describing the behavior when a server loses access to the customer-managed TDE protector.

T+0 — AKV Access Lost

Primary MI loses access to Azure Key Vault A. The TDE protector key can no longer be retrieved. The encryption cache still holds the key temporarily.

T+10 minutes — Connections Denied

The primary database starts denying all connections with an error message indicating the TDE protector is inaccessible. The database state begins transitioning to Inaccessible.

Behavior varies by error type:

  • Transient / 5XX errors (networking outage): A 24-hour buffer is applied before the database is moved to Inaccessible state. Databases may come back online automatically.
  • Persistent / 4XX errors (permission revoked, key deleted): The database is moved to Inaccessible state after 30 minutes.
T+30 minutes — Critical Threshold (4XX Errors)

If AKV access has not been restored within 30 minutes (for 4XX errors), the database enters the Inaccessible state. Automatic healing is no longer possible.

⚠️ Auto-Heal Cutoff If key access is restored within 30 minutes, the database automatically heals within the subsequent hour. After 30 minutes, manual intervention through the Azure portal is required, and recovery time depends on database size.
T > 30 minutes — Control Plane Breaks FOG Link

The Azure SQL control plane detects the Inaccessible state and breaks the Failover Group link. The database is converted into a standalone database.

Once the link is broken, the following are permanently lost:

  • Failover Group configuration
  • Point-in-time restore (PITR) history
  • Tags and database-level settings
  • Elastic pool configurations, read scale, auto-pause settings
  • Long-term retention policy

Force Failover — Time Window

Force Failover must be executed before the control plane breaks the FOG link. Once the link is broken, there is no Failover Group to fail over.

Available Time Window

Error TypeTime to InaccessibleForce Failover WindowRecommendation
4XX Errors
(Permission revoked, key deleted, identity deleted)		 30 minutes ≤ 30 minutes Act immediately — execute Force Failover within minutes
5XX Errors
(Network outage, transient AKV failure)		 Up to 24 hours (buffer) Up to 24 hours Monitor closely; may auto-recover. Force failover if not resolved within your RTO
⚠️ Important: The Exact Moment the Control Plane Breaks the Link Is Not Precisely Documented Documentation states the link is broken when the database "becomes inaccessible." The 30-minute threshold for 4XX errors is the documented cutoff for automatic healing. The actual link-break timing may vary. Do not wait — initiate Force Failover as early as possible.

Decision Tree

Primary lost AKV access? │ ├─ Can restore AKV access within 30 min? │ ├─ YES → Restore access → Auto-heal (within ~1 hour) │ └─ NO → Proceed to Force Failover ↓ │ ├─ Is FOG link still intact? │ ├─ YES → Execute Force Failover NOW → Secondary promoted to R/W Primary │ └─ NO → Force Failover NOT possible → See Section 9 (Recovery) │ └─ After successful Force Failover: ├─ Old Secondary (now Primary) serves R/W via AKV-B ✅ ├─ Old Primary becomes new Secondary after AKV-A restored └─ Failback when ready

Force Failover — Methods

Force Failover for MI Failover Groups can be executed through multiple Azure management interfaces. It is NOT possible via T-SQL — MI FOG is an Azure control-plane resource, not an AG managed within the SQL engine.

🚨 T-SQL Is NOT Supported Unlike SQL Server Availability Groups (ALTER AVAILABILITY GROUP ... FORCE_FAILOVER_ALLOW_DATA_LOSS), MI Failover Groups cannot be failed over using T-SQL commands. You must use Azure management tools.

Method 1: Azure Portal

  1. Navigate to the Secondary Managed Instance in the Azure portal.
  2. Under Data management, select Failover groups.
  3. Click Failover from the command bar.
  4. Confirm the warning about TDS session disconnection and potential data loss.
  5. Wait for the roles to switch — the secondary becomes the new primary.

Method 2: Azure PowerShell

# Execute on the Secondary MI's resource group and location
Switch-AzSqlDatabaseInstanceFailoverGroup `
    -ResourceGroupName "<Secondary-ResourceGroup>" `
    -Location "<Secondary-Region>" `
    -Name "<FailoverGroupName>" `
    -AllowDataLoss    # This flag triggers FORCE failover

Method 3: Azure CLI

az sql instance-failover-group set-primary \
    --name "<FailoverGroupName>" \
    --resource-group "<Secondary-ResourceGroup>" \
    --location "<Secondary-Region>" \
    --allow-data-loss    # This flag triggers FORCE failover

Method 4: REST API

POST https://management.azure.com/subscriptions/{subscriptionId}
    /resourceGroups/{resourceGroupName}
    /providers/Microsoft.Sql/locations/{locationName}
    /instanceFailoverGroups/{failoverGroupName}
    /forceFailoverAllowDataLoss?api-version=2023-05-01-preview

Method 5: .NET SDK

// Using Microsoft.Azure.Management.Sql
await sqlClient.InstanceFailoverGroups
    .ForceFailoverAllowDataLossAsync(
        resourceGroupName,
        locationName,
        failoverGroupName
    );

💡 Recommendation: Pre-build an Emergency Failover Script Given the narrow time window (≤30 minutes for 4XX errors), it is critical to have a pre-tested PowerShell or CLI script ready. Relying on the Portal UI during an emergency wastes precious minutes.

Emergency Script Template (PowerShell)

# emergency-fog-failover.ps1
# Pre-configure these values for your environment
param(
    [string]$FOGName         = "myFailoverGroup",
    [string]$SecondaryRG     = "rg-secondary-eastasia",
    [string]$SecondaryRegion = "eastasia"
)

Write-Host "⚠️  Initiating FORCE FAILOVER for FOG: $FOGName" -ForegroundColor Yellow
Write-Host "    Target: $SecondaryRG / $SecondaryRegion"
Write-Host "    WARNING: This may cause data loss!" -ForegroundColor Red

$confirm = Read-Host "Type 'FAILOVER' to confirm"
if ($confirm -ne "FAILOVER") {
    Write-Host "Aborted."; return
}

$stopwatch = [System.Diagnostics.Stopwatch]::StartNew()

Switch-AzSqlDatabaseInstanceFailoverGroup `
    -ResourceGroupName $SecondaryRG `
    -Location $SecondaryRegion `
    -Name $FOGName `
    -AllowDataLoss

$stopwatch.Stop()
Write-Host "✅ Force Failover completed in $($stopwatch.Elapsed.TotalSeconds) seconds" -ForegroundColor Green

Force Failover — Data Loss Implications

MI Failover Groups use asynchronous replication. A Force Failover promotes the secondary without waiting for all data to be synchronized from the primary.

What Data Can Be Lost?

Data CategoryAt Risk?Details
Committed transactions not yet replicated Yes Any transactions committed on the primary but not yet propagated to the secondary via async replication will be lost.
In-flight transactions Yes Uncommitted transactions on the primary are rolled back and not present on the secondary.
Data already replicated to secondary No All data that was successfully replicated is safe on the secondary.

How Much Data Loss?

The amount of data loss depends on the replication lag at the moment the primary became unavailable. Replication lag is affected by:

  • Network latency between the two regions
  • Transaction volume on the primary
  • Database size and I/O characteristics
ℹ️ Monitoring Replication Lag Use sys.dm_geo_replication_link_status to monitor the replication lag (replication_lag_sec) during normal operations. This gives you an estimate of potential data loss in a force failover scenario. For MI, you can also monitor via Azure Monitor metrics.

After Force Failover Completes

  • The old secondary becomes the new primary (read-write) using AKV-B.
  • The old primary will attempt to reconnect as the new secondary once its AKV-A access is restored.
  • A failback can be performed when the original primary is healthy again.
  • Any data that was on the old primary but not replicated is permanently lost — it cannot be recovered after the failover.

When Force Failover Is No Longer Possible

Force Failover becomes impossible when the Azure SQL control plane has already broken the FOG link.

Conditions That Prevent Force Failover

ConditionWhy It Prevents Failover
FOG link already broken by control plane There is no Failover Group to fail over. Both instances are now standalone.
FOG manually deleted The Failover Group resource no longer exists.
Secondary MI also lost its AKV access Both instances are Inaccessible — neither can serve as primary.
Secondary MI is offline / unavailable The target for failover does not exist or is not reachable.
🚨 Once the Link Is Broken, You Cannot:
  • Execute any Force Failover (no FOG exists)
  • Promote the secondary to read-write via FOG commands
  • Use FOG DNS listeners (they no longer resolve)
  • Recover the old PITR history or tags

How to Know If the Link Is Already Broken

  • Azure Portal: Navigate to the MI → Failover Groups. If no FOG is listed, the link has been dissolved.
  • PowerShell: Get-AzSqlDatabaseInstanceFailoverGroup will return an error or empty result if the FOG no longer exists.
  • Azure Resource Health: Check for resource health alerts indicating the FOG link was broken.
  • Activity Log: Look for control plane operations related to FOG deletion or link break.

Secondary Status After FOG Link Break

This is the most critical ambiguity in the current documentation. When the control plane breaks the FOG link because the primary became inaccessible, the secondary's behavior depends on several factors.

What the Documentation Says

📖 Official Documentation Quote "When a database is part of active geo-replication or failover groups and becomes inaccessible, the SQL control plane breaks the link and converts the database into a standalone database. After fixing the key permissions, the primary database can typically be brought back online. The secondary database can't be brought back online because Azure SQL doesn't take full backups for secondary databases by design. The recommendation is to drop the secondary databases and re-establish the link."

— Azure SQL TDE with customer-managed key — Geo-DR section

Analysis: Does This Apply When Secondary Has Its Own Working AKV?

The documentation statement "the secondary database can't be brought back online" is written in the context of TDE inaccessibility scenarios. There is an important ambiguity:

Interpretation A — Secondary Also Inaccessible

The statement may primarily apply when both primary and secondary lost AKV access (e.g., shared AKV in the primary region fails). In this case, the secondary truly can't be brought online because it has no key to decrypt.

Likely the documented scenario

Interpretation B — Universal Limitation

The statement may apply universally: regardless of whether the secondary has its own AKV, the control plane's link-break mechanism puts the secondary in a state that cannot be promoted to read-write without a full backup (which doesn't exist).

Cannot be ruled out

Secondary Access State Summary

ScenarioSecondary DB StateRead AccessWrite Access
FOG link intact, before link break Geo-Secondary (Read-Only) Yes No
After link break — secondary has AKV Standalone (uncertain state) Uncertain No
After link break — secondary lost AKV too Inaccessible No No
Manual Stop Replication (normal scenario) Standalone Read-Write Yes Yes
🚨 Key Takeaway The documentation does not guarantee that a secondary with a working AKV will be promoted to read-write after a control-plane-initiated link break. The recommended path is to drop the secondary and re-establish the link after fixing the primary. Do NOT rely on the secondary becoming independently usable after a link break.

Recovery Procedures

Scenario A: AKV Access Restored Within 30 Minutes

Best Outcome — Automatic Healing

  1. Restore AKV access for the primary (fix permissions, restore key, fix networking).
  2. The database automatically heals within approximately 1 hour.
  3. FOG link remains intact — no action needed.
  4. Validate all database settings after recovery.

Scenario B: Force Failover Executed Successfully (Within Time Window)

Controlled Recovery — Some Data Loss Possible

  1. Force Failover is executed to the secondary in Region B.
  2. Secondary is promoted to primary (read-write) using AKV-B. ✅
  3. Applications reconnect via the FOG R/W listener (DNS automatically updated).
  4. Restore AKV-A access on the old primary.
  5. The old primary automatically becomes the new secondary after reconnection.
  6. (Optional) Perform a failback to restore original topology.

Scenario C: FOG Link Already Broken (Force Failover Not Possible)

Worst Outcome — Full Rebuild Required

  1. Fix Primary AKV access:
    • Restore key vault permissions (get, wrapKey, unwrapKey)
    • Restore deleted keys (if applicable — use AKV soft-delete / purge protection)
    • Fix networking (NSG rules, private endpoint configuration)
    • Restore managed identity in Microsoft Entra ID
  2. Revalidate TDE key access via the Azure portal (manual procedure required after >30 min outage).
  3. Bring primary database back online — this may take time depending on database size.
  4. Validate all server and database settings — FOG config, PITR, tags, retention policies are lost and must be reconfigured.
  5. Drop the old secondary databases — per Microsoft recommendation, they cannot be brought back online.
  6. Re-create the Failover Group from scratch:
    • Create a new FOG linking Primary MI → Secondary MI
    • Databases will be re-seeded to the secondary (time depends on data size)
    • Reconfigure failover policy, grace period, DNS listeners
⚠️ Recovery Time for Scenario C Full recovery in Scenario C can take hours to days depending on:
  • Time to restore AKV access
  • Database size (manual recovery procedure)
  • Re-seeding time for the secondary
  • Reconfiguration of all lost settings

Recovery Comparison

ScenarioData LossDowntimeManual EffortSettings Lost

Monitoring & Automation Recommendations

Given the narrow 30-minute window for 4XX errors, proactive monitoring is essential.

Monitoring Setup

What to MonitorToolAlert Threshold
AKV access from MI Azure Key Vault Diagnostics + Azure Monitor Any 4XX error from MI's managed identity
Database accessibility Azure Resource Health State change to "Degraded" or "Unavailable"
FOG replication lag Azure Monitor Metrics / sys.dm_geo_replication_link_status Lag exceeding your RPO threshold
TDE protector status Azure Activity Log Any operation modifying key permissions
MI connectivity to AKV Scheduled connectivity test from MI TCP test to AKV endpoint on port 443 fails

Automation Best Practices

  • Azure Action Groups: Configure alerts to notify on-call engineers immediately when AKV access issues are detected.
  • Pre-tested Failover Scripts: Maintain PowerShell/CLI scripts with pre-configured parameters for emergency failover. Test quarterly.
  • Runbook Documentation: Document the exact steps, credentials, and approvals needed for force failover so that on-call engineers can execute within minutes.
  • AKV Soft-Delete & Purge Protection: Always enable these on key vaults used for TDE to prevent accidental permanent key deletion.
  • Regular DR Drills: Perform planned failover tests (without -AllowDataLoss) at least quarterly to validate the FOG configuration and team readiness.

AKV Protection Checklist

🔒 Key Vault Configuration

  • ✅ Soft-delete enabled (default since 2020)
  • ✅ Purge protection enabled
  • ✅ Access policies grant get, wrapKey, unwrapKey to MI's managed identity
  • ✅ Network rules allow MI subnet access
  • ✅ Diagnostic logging enabled

🛡️ Operational Safeguards

  • ✅ Azure RBAC restricts who can modify AKV permissions
  • ✅ Azure Policy prevents accidental AKV deletion
  • ✅ Alerting on permission changes via Activity Log
  • ✅ Key material synchronized across AKV-A and AKV-B
  • ✅ Emergency failover script tested and accessible
