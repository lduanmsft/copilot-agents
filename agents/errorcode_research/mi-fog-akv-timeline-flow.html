MI FOG + TDE/AKV Key Loss — Timeline Flow

Scenario: Primary MI loses access to AKV key (4XX error — permission/key issue)
✅ Updated April 16, 2026 — Open questions resolved via live testing

Scenario A: Shared AKV (Both MI access the same Key Vault)

If the AKV key itself is deleted/expired/disabled, both Primary and Secondary lose access.

Scenario B: Separate AKVs (Each MI has its own Key Vault)

Only Primary loses AKV access. Secondary's AKV is unaffected.

Scenario C: Shared AKV — Only Primary's Network to AKV is Broken

AKV key is fine. Only Primary MI cannot reach AKV (NSG/DNS/firewall issue). Secondary can still reach AKV.

Key Findings

1. Only TDE-protected databases become inaccessible. The non-TDE database (test1) remained online on primary throughout the entire test.
2. Secondary becomes standalone read-write after FOG drops. Both databases (TDE and non-TDE) become fully writable on the secondary MI. This is consistent with the documented behavior when deleting a FOG: "The secondary databases are disconnected from the primaries and become read-write databases."
3. Secondary starts auto-backup immediately. MI-2 begins taking automatic backups for both databases as soon as they become standalone, establishing a new PITR baseline.
4. Separate AKVs provide a significant safety net. Even if you miss the 30-minute force failover window, secondary databases survive as standalone read-write copies with their own AKV. With shared AKV + key-level failure, both instances go down with no recovery path until the key is restored.
