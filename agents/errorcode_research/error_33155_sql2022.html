Error Trigger Code — FedAuth Disconnect Handling

else if (pNetConn->FInFedAuthReady())
{
    // If in FedAuthReady, release a ref on the connection,
    // since we would have added it when sending FedAuthInfoToken.

    if (event != esetConnectFedAuth && event != esetDisconnect)
    {
        // A bad TDS token was sent by the client when the server
        // is waiting for federated authentication token.
        // Disconnect the mis-behaving client.
        ReportLoginFailureByNetConn(pNetConn,
            EX_NUMBER(SECURITYERR7, SEC_FEDAUTHREADY_BAD_TDS_TOKEN), event);
        event = esetDisconnect;
    }
    else if (event == esetDisconnect)
    {
        // A disconnect event was raised when server is waiting for
        // federated authentication token from the client.
        // This could be either due to client close or the server
        // timing out waiting for the token and closing the connection.
        ReportLoginFailureByNetConn(pNetConn,
            EX_NUMBER(SECURITYERR7, SEC_FEDAUTHREADY_DISCONNECT), 1);
    }
}
else if (event == esetConnectFedAuth && !pNetConn->FInFedAuthReady())
{
    // We are receiving Federated Authentication Token when not expecting it.
    // Report and disconnect.
    ReportLoginFailureByNetConn(pNetConn,
        EX_NUMBER(SECURITYERR7, SEC_FEDAUTHTOKEN_RECEIVED_NOT_IN_FEDAUTHREADY), 1);
    event = esetDisconnect;
}

Key Observations

  • Error 33155 is raised with state = 1 always.
  • It occurs exclusively when pNetConn->FInFedAuthReady() returns true AND an esetDisconnect event is received.
  • The FInFedAuthReady() state means the server has already sent the FedAuthInfoToken to the client and is awaiting the client's federated authentication response.
  • Two root causes: (1) Client closed the connection voluntarily, or (2) Server timeout expired while waiting for the FedAuth token.
  • Related sibling errors in the same code block:
    • 33154 (SEC_FEDAUTHREADY_BAD_TDS_TOKEN) — wrong TDS token during FedAuth wait
    • 33156 (SEC_FEDAUTHTOKEN_RECEIVED_NOT_IN_FEDAUTHREADY) — FedAuth token received unexpectedly
📄 sniinteg.cpp — ReportLoginFailureByNetConn()

File: /Sql/Ntdbms/tds/src/sniinteg.cpp

Purpose: Reports login failure to event log, trace, and connectivity ring buffer.

void ReportLoginFailureByNetConn(
    __in CNetConnection* pNetConn,  // CNetConnection for spid and SNI_Conn for host info
    _SQL_ERROR_CODE_ int MesgID,   // message number
    __in int state)                 // unique state
{
    Assert(pNetConn);
    pNetConn->TraceError(0, MesgID, state);

    CLoginPeerInfo lci(pNetConn->NetlibEP());
    bool fIsDuplicated = false;

    pNetConn->GetLock();
    if (!pNetConn->FClosed())
    {
        fIsDuplicated = pNetConn->PPhyConn()->IsDuplicatedConnection();
    }
    pNetConn->ReleaseLock();

    // Delegate to the appropriate login failure callback function
    g_pTdsHandle->m_tdscallbacks.GetLoginFailureFn(pNetConn)(
        pNetConn->PIBatch(),
        0 /*cbLoginName*/,
        NULL /*pwsLoginName*/,
        MesgID,
        fIsDuplicated,
        pNetConn->NetlibEP(),
        state,
        nullptr /*pErrorContext*/,
        lci.Cb(), lci.Pwch());
}

5. Connectivity Ring Buffer

The error corrective action explicitly states: "Please refer to other errors or connectivity ring buffer events for further root cause."

The ConnectivityTraceManager class (in sniinteg.cpp) stores records into RING_BUFFER_CONNECTIVITY via StoreRecord().

Query Connectivity Ring Buffer

SELECT
    record.value('(Record/@id)[1]', 'int') AS RecordId,
    record.value('(Record/@time)[1]', 'bigint') AS RecordTime,
    record.value('(Record/ConnectivityTraceRecord/RecordType)[1]', 'varchar(50)') AS RecordType,
    record.value('(Record/ConnectivityTraceRecord/TdsError)[1]', 'int') AS TdsError,
    record.value('(Record/ConnectivityTraceRecord/State)[1]', 'int') AS State,
    record.value('(Record/ConnectivityTraceRecord/RemoteHost)[1]', 'varchar(50)') AS RemoteHost,
    record.value('(Record/ConnectivityTraceRecord/RemotePort)[1]', 'int') AS RemotePort
FROM (
    SELECT CAST(record AS XML) AS record
    FROM sys.dm_os_ring_buffers
    WHERE ring_buffer_type = 'RING_BUFFER_CONNECTIVITY'
) AS rb
WHERE record.value('(Record/ConnectivityTraceRecord/TdsError)[1]', 'int') = 33155
ORDER BY RecordId DESC;

Context: FedAuth Login Flow

Error 33155 occurs during the Federated Authentication (FedAuth) login flow, which is used for Azure Active Directory / Microsoft Entra ID authentication in Azure SQL Database and SQL Server:

  1. Client initiates TDS connection and sends prelogin/login packets
  2. Server determines FedAuth is needed and sends FedAuthInfoToken (containing STS URL and SPN)
  3. Server enters FInFedAuthReady state and waits for client response
  4. Client should acquire token from Azure AD/Entra ID and send FedAuthToken
  5. ⚡ If the client disconnects or the server times out during step 4 → Error 33155

Investigation Checklist

#StepDetails
1 Check Connectivity Ring Buffer Query sys.dm_os_ring_buffers with RING_BUFFER_CONNECTIVITY for correlated errors around the same timestamp.
2 Check for AAD/Entra ID outages Error typically indicates an issue in the ADAL / Entra ID layer, not SQL Server itself. Check Azure status page.
3 Check for network issues Client disconnect may be caused by network interruptions between client → Azure AD or client → SQL Server.
4 Check for correlated error 18456 Error 33155 frequently co-occurs with error 18456 during AAD issues. Check MonLogin for both.
5 Check client timeout settings Client may have a connection timeout shorter than the time needed to acquire an AAD token. Increase ConnectTimeout in the connection string.
6 Refer to TSG Error 33155 - Is Fed Auth Login TSG

⚠️ Note: No specific "fix" workaround is available because this error is a symptom of an upstream issue (client disconnect, AAD timeout, or network failure). The root cause must be investigated using the correlated errors and ring buffer events.
