//-----------------------------------------------------------------------------
// HadrAGListenerNotifier::ReportFailureToUser
//
// Description:
//   Report a TDS listener binding failure to the user via ex_callprint_error
//
// Exceptions:
//   None
//
void
HadrAGListenerNotifier::ReportFailureToUser () const
{
    if (!m_parentListener.IsNull () &&
        (m_parentListener.GetTcpPort () > 0) &&
        ((WSAEADDRINUSE == m_dwError) || (WSAEACCES == m_dwError)))
    {
        ex_callprint_error (
            EX_NUMBER (HADR_ERROR2, HADR_DDL_LISTENER_TCP_PORT_IN_USE_FAILURE),
            EX_USER,
            1,
            m_parentListener.GetDnsName ().ByteCount (),
            m_parentListener.GetDnsName ().Wsz (),
            m_parentListener.GetTcpPort ());
    }
}

💡 Analysis

Purpose

HadrAGListenerNotifier::ReportFailureToUser() reports a TCP listener binding failure back to the DDL-executing session after the TDS Listener Synchronizer has attempted (and failed) to bind the AG listener's TCP port.

How the Error is Raised

  • The error is raised via ex_callprint_error() — a non-fatal print-to-client mechanism (not ex_raise)
  • It only fires when all three conditions are true:
    1. m_parentListener is not null (a listener was configured)
    2. GetTcpPort() > 0 (a non-zero TCP port was specified)
    3. m_dwError is WSAEADDRINUSE (10048) or WSAEACCES (10013)
  • Error inserts: DNS name (%.*ls) and port number (%d)

Error Handling Behavior

  • Non-Fatal — Uses ex_callprint_error (prints message to client, does NOT abort the DDL)
  • The AG listener DDL command itself succeeds (metadata is committed), but the listener cannot actually accept connections on that port
  • The asynchronous architecture means:
    • DDL thread waits on PWAIT_HADR_TDS_LISTENER_SYNC_PROCESSING for TDS synchronizer
    • TDS synchronizer attempts to bind the port asynchronously
    • On failure, it signals back via Signal(dwError, listener)
    • DDL thread wakes up and calls ReportFailureToUser()
  • No trace flag bypass exists for this error

Root Cause

Windows ErrorCodeMeaning
WSAEADDRINUSE 10048 The port is already bound by another process or another AG listener
WSAEACCES 10013 Permission denied — SQL Server service account lacks permission to bind the port, or the port is in an exclusive-use range

Common scenarios:

  • Another application (e.g., IIS, another SQL instance) is already listening on the same port
  • A previous AG listener on the same port wasn't fully cleaned up
  • The SQL Server service account doesn't have privileges to bind to a low-numbered port (<1024)
  • Windows Firewall or security policy blocking port binding

-- Check which ports AG listeners are configured on
SELECT ag.name AS ag_name,
       agl.dns_name,
       aglip.ip_address,
       agl.port
FROM sys.availability_group_listeners agl
JOIN sys.availability_groups ag ON ag.group_id = agl.group_id
LEFT JOIN sys.availability_group_listener_ip_addresses aglip
    ON aglip.listener_id = agl.listener_id
ORDER BY agl.port;

📝 Resolution Steps

  1. Identify the conflicting process — Use netstat -ano | findstr :<port> or Get-NetTCPConnection to find what's using the port
  2. Free the port — Stop the conflicting application, or move the AG listener to a different port: 
    ALTER AVAILABILITY GROUP [YourAG]
MODIFY LISTENER N'YourListener' (PORT = new_port);
  3. If WSAEACCES (10013) — Verify the SQL Server service account has permission to bind the port. For ports <1024, the service may need elevated privileges or use netsh http add urlacl
  4. Restart the listener — After freeing the port, take the AG listener offline and online in WSFC, or restart the AG resource group
